Overview

The admin portal is the control surface for monitoring system totals, refreshing synced records, managing user access, and controlling device availability.

Dashboard
View totals for devices, active devices, users, and blocked users.
Devices
Restrict, allow, or delete device records from the backend.
Users
Block, unblock, edit display names, generate reset links, or delete user accounts.
Sync
Force a backend refresh when recently changed records are not visible yet.
2

Login and Access

Logging In
  • 1
    Open the portal

    Open the admin portal URL in a browser.

  • 2
    Enter credentials

    Type the admin username and password configured on the EC2 server.

  • 3
    Open the portal

    After successful authentication, the portal loads the Dashboard tab.

Logging Out
  • 1
    Use logout

    Click the logout action in the portal.

  • 2
    Session clears

    The browser session is cleared and the login screen is shown again.

Credential sourcePortal credentials are read from environment variables on the server, not from a user table inside the UI.

Sync Data

Use the Sync Data button when you want a fresh backend read immediately instead of waiting for background refresh.

  • 1
    Click Sync Data

    The portal starts a backend refresh request.

  • 2
    Wait for status

    The UI shows a temporary sync-in-progress message.

  • 3
    Review results

    When finished, the portal reloads dashboard totals plus the current users and devices lists.

When to use itUse sync after a recent app sign-up, unblock action, device change, or backend reconciliation event.

Managing Devices

The Devices tab lists each device ID, name, location, status, last seen timestamp, and available actions.

Statuses
Active Restricted
Available Actions
  • 1
    Restrict or Allow

    Click Restrict to disable an active device, or Allow to restore a restricted device.

  • 2
    Delete Device

    Click Delete and confirm when the device record should be removed completely.

Delete carefullyDeleting a device is more than a temporary restriction. Use restriction for reversible access control and deletion only for permanent removal.
6

Managing Users

The Users tab lists each user ID, email, display name, authentication type, account status, last login, and row-level actions.

FieldMeaning
Auth TypeGoogle or Email/Password
StatusActive means allowed, Blocked means denied
Last LoginLast known sign-in time or Never
Actions
  • 1
    Block or Unblock

    Use these actions to deny or restore app access. Email/password sign-up users are initially blocked and must be unblocked before first login.

  • 2
    Edit Display Name

    Available for eligible email/password users. A browser prompt asks for the updated display name.

  • 3
    Forgot Password

    Available for eligible email/password users. The portal generates a password reset link that can be copied and shared.

  • 4
    Delete User

    Deletes the user from EC2-backed records. Use only for permanent account removal.

Forgot Password limitationsThe reset-link action is not shown for Google users and is not shown for Gmail-address users on the password flow.
7

Email/Password Approval Flow

Non-Gmail credential users created from the mobile app require explicit admin approval before first login.

  • 1
    User signs up in the app

    The app creates the Firebase credential and sends the record to the backend.

  • 2
    Backend stores the user as blocked

    The account appears in the Users tab with blocked status by default.

  • 3
    Admin reviews the row

    Find the user in the Users tab and verify the email and display name.

  • 4
    Click Unblock

    This activates the account for sign-in.

  • 5
    User signs in

    The user can then log in with the same email and password from the app.

Expected state before approvalIf the user says sign-up succeeded but login is denied, the Users tab should still show that account as Blocked until you click Unblock.
8

Troubleshooting

Login fails

Verify the admin username and password in the EC2 environment file and restart the PM2 process if credentials changed recently.

User remains blocked

Use Sync Data, refresh the Users tab, and verify the status actually changed from Blocked to Active.

Reset link option is missing

That action only appears for eligible non-Gmail email/password users.

Data looks stale

The portal auto-refreshes visible tab data roughly every 10 seconds, but you can still use Sync Data for an immediate backend refresh.
